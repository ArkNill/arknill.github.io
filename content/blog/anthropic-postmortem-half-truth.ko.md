4월 23일, Anthropic은 3월 4일부터 4월 20일까지 Claude Code를 저하시킨 세 가지 제품 레이어 버그를 인정하는 [포스트모템](https://www.anthropic.com/engineering/april-23-postmortem)을 발표했습니다.
그들의 프레이밍은 이렇습니다: 모델 가중치는 변경되지 않았고, 하네스 버그를 수정했으며, 문제는 해결되었다.

**세 가지 버그는 실재합니다. 그 영향도 실재했습니다. 수정도 실재했습니다.**

그러나 이 포스트모템은 진실의 절반만 말하는, 신중하게 범위가 설정된 문서입니다.
나머지 절반은 다음과 같습니다.

---

## Anthropic이 인정한 것 (정확한 부분)

| 버그 | 도입일 | 수정일 | 기간 |
|-----|-----------|-------|----------|
| Effort `high` → `medium` | March 4 (v2.1.68) | April 21 (v2.1.117) | **48일** |
| Thinking cache를 매 턴 초기화 | March 26 (v2.1.85) | April 10 (v2.1.101) | 15일 |
| "≤25 words" 시스템 프롬프트 | April 16 (v2.1.111) | April 20 (v2.1.116) | 4일 |

세 가지 모두 제품 레이어 이슈입니다 — API 파라미터와 시스템 프롬프트의 문제이지, 모델 가중치의 문제가 아닙니다.
복합 효과는 다음과 같습니다: effort 감소는 사고 깊이를 줄였고, thinking cache 초기화는 세션 메모리를 파괴했으며, 단어 제한은 출력을 절단했습니다.
이 세 가지가 합쳐져 Claude가 현저히 멍청해진 것처럼 보이게 만들었습니다.

이 설명은 기술적으로 정확합니다.
세 가지 버그를 동시에 경험한 사용자라면 보고된 것과 정확히 일치하는 성능 저하를 체감했을 것입니다.

이 부분은 수용합니다.

## 전략적 범위 설정

이 포스트모템은 "4.6 사용자가 성능 저하를 느낀 이유"를 다룹니다.
그러나 별도의, 동시에 존재하는 불만 흐름은 다루지 않습니다: **"4.7이 4.6보다 나쁘다."**

이것들은 하네스 버그와 무관한 모델 수준의 회귀입니다:

| Opus 4.7 모델 이슈 | 근거 | 하네스 버그? |
|---------------------|----------|:---:|
| Tokenizer +35% (CJK에서 더 심함) | [공식 문서](https://platform.claude.com/docs/en/about-claude/models/whats-new-claude-4-7): "1.0x to 1.35x more tokens" | 아니오 — 아키텍처 변경 |
| Long-context retrieval: 91.9% → 59.2% (256K) | Anthropic 자체 system card (MRCR v2) | 아니오 — 모델 성능 |
| Long-context retrieval: 78.3% → 32.2% (1M) | Anthropic 자체 system card (MRCR v2) | 아니오 — 모델 성능 |
| 지시를 문자 그대로 따르는 행동 | [What's New](https://platform.claude.com/docs/en/about-claude/models/whats-new-claude-4-7): "follows instructions more literally" | 아니오 — 학습 변경 |
| BrowseComp: 84.0% → 79.6% | Anthropic 벤치마크 | 아니오 — 모델 회귀 |
| 긴 페이로드에서 XML/JSON 혼합 ([#49747](https://github.com/anthropics/claude-code/issues/49747)) | 디코더 수준 포맷 전환 | 아니오 — 모델 동작 |
| 안전성 과잉 거부 ([#49751](https://github.com/anthropics/claude-code/issues/49751)) | 구조 생물학이 4.7에서 차단, 4.6에서는 정상 작동 | 아니오 — RLHF/안전 레이어 |

포스트모템은 이 중 어느 것도 언급하지 않습니다.
단 한 줄도 없습니다.
서사는 이렇습니다: "세 가지 버그를 수정했고, Claude는 정상으로 돌아왔습니다."
그러나 4.7로 업그레이드한 사용자들은 세 가지 버그 수정으로 해결되지 않는 별도의 문제들을 겪고 있습니다.

**이 포스트모템은 Stream A("4.6이 멍청해졌다")를 해결하면서 Stream B("4.7은 회귀다")가 존재하지 않는 것처럼 행동합니다.**

## 비용 우연의 일치

인정된 세 가지 버그는 공통된 방향을 공유합니다:

| 버그 | 사용자 영향 | Anthropic 서빙 비용 |
|-----|------------|----------------------|
| Effort `high` → `medium` | 품질 저하 | **컴퓨팅 ↓↓** |
| Thinking cache를 매 턴 초기화 | 컨텍스트 유실, 반복 | **Prefill ↓↓** (캐시할 것이 줄어듦) |
| "≤25 words" 출력 제한 | 정보량 감소 | **출력 토큰 ↓↓** |

세 가지 변경 모두 독립적으로 Anthropic의 요청당 서빙 비용을 절감합니다.
"UI 프리징 수정," "캐싱 최적화," "장황함 감소"로 프레이밍되었습니다.

의도를 증명할 수는 없습니다. 그러나:
- 3/3 변경이 우연히 비용 절감과 일치할 확률은 낮습니다
- 세 가지 중 두 가지는 **CHANGELOG에 기록되지 않았습니다** — 논란이 될 것이라는 조직적 인식을 시사합니다
- Effort 변경은 기록되었지만 제품 개선("속도와 철저함 사이의 최적점")으로 프레이밍되었고, 비용 동기의 트레이드오프로는 절대 표현되지 않았습니다

비용 동기 없이 순수한 실수였다면, 왜 두 가지를 changelog에서 완전히 숨겼을까요?

## CHANGELOG 공백이 진짜 이야기입니다

버그는 잠시 잊으십시오. 구조적 발견은 이것입니다:

> **Anthropic은 공개 기록 없이 Claude Code의 동작에 영향을 미치는 변경을 할 수 있고, 실제로 하고 있습니다.**

Thinking cache 버그는 3월 26일에 도입되어 4월 10일에 수정되었습니다.
두 이벤트 모두 CHANGELOG에 나타나지 않습니다.
3,285줄 전체를 검색했습니다 — "thinking cache," "cache clear," "thinking prun"이라는 용어는 **어디에도 없습니다**.

"≤25 words" 시스템 프롬프트는 4월 16일에 추가되어 4월 20일에 되돌려졌습니다.
두 이벤트 모두 CHANGELOG에 나타나지 않습니다.
"25 words"와 "length limit"이라는 용어는 **어디에도 없습니다**.

시스템 프롬프트 변경은 구조적으로 보이지 않습니다. 사용자는 다음을 할 수 없습니다:
- 자신의 버전에서 어떤 시스템 프롬프트가 활성화되어 있는지 확인
- 버전 간 시스템 프롬프트 차이 비교
- 시스템 프롬프트 변경이 언제 경험을 저하시켰는지 인지
- "모델이 멍청해졌다"와 "보이지 않는 지시가 제한하고 있다"를 구별

포스트모템의 개선 계획은 "안정화 기간"과 "점진적 롤아웃"을 언급하지만 — 동작에 영향을 미치는 변경에 대한 **의무적 문서화**에 대해서는 아무것도 말하지 않습니다.
구조적 불투명성은 그대로 유지됩니다.

## 포스트모템 이후: 수정되지 않았습니다

포스트모템의 서사는 "세 가지 버그, 4월 20일 기준 모두 수정됨"입니다. 4월 22-24일에 v2.1.117-119에서 제출된 이슈들은 그렇지 않음을 보여줍니다:

| 이슈 | 문제 | Anthropic 대응 |
|-------|---------|-------------------|
| [#52502](https://github.com/anthropics/claude-code/issues/52502) | Subagent `model: haiku` 지정이 무시됨 — 모든 것이 Opus로 실행 ($10.87 vs $0.0005) | 없음 |
| [#52534](https://github.com/anthropics/claude-code/issues/52534) | `effortLevel` 설정이 `unpinOpus47LaunchEffort` 플래그에 의해 오버라이드 — 프로그래밍 제어 불가 | 없음 |
| [#52522](https://github.com/anthropics/claude-code/issues/52522) | Auto-compact 임계값이 200K→1M으로 변경 — 하룻밤 사이 5배 토큰 사용량 | 봇: "중복" |
| [#52228](https://github.com/anthropics/claude-code/issues/52228) | 모델이 "Human:" 프롬프트를 날조하고, 일방적 워크스테이션 작업과 함께 자기 대화 시작 | 없음 |

이것들은 엣지 케이스가 아닙니다.
#52502는 직접적인 재정적 영향이 있는 과금 이슈입니다.
#52534는 effort 설정이 작동하지 않는다는 의미입니다.
#52522는 "수정"이 기존 사용자에게 5배 비용 증가를 초래했다는 의미입니다.

"모두 수정됨" 프레이밍은 새로운 이슈가 이를 부정하기까지 약 48시간 지속되었습니다.

## 대신 이랬다면 존중했을 것입니다

포스트모템이 이렇게 말했다면:

> *"4.6 성능 저하 보고를 설명하는 세 가지 하네스 버그를 발견했습니다. 수정했습니다. 별도로, Opus 4.7에는 알려진 트레이드오프(tokenizer 효율, long-context retrieval, instruction following 동작)가 있으며 작업 중입니다. 또한 CHANGELOG가 시스템 프롬프트 변경을 다루지 않는다는 점을 인정하며, 이를 개선할 방법을 검토하고 있습니다."*

그것이 완전한 포스트모템이었을 것입니다.
그들이 발표한 것은 **방어 가능한 부분집합에 대한 인시던트 대응** — PR 성명의 역할을 하는 엔지니어링 블로그 포스트입니다.

## 제 입장

저는 4월 15일부터 고정된 버전(v2.1.109)에서 Opus 4.6으로 Claude Code를 운영하고 있습니다.
포스트모템의 세 가지 버그는:
- 버그 #1 (effort): v2.1.68+에 영향, 제 버전은 3월 4일~4월 21일 노출 (**48일**)
- 버그 #2 (thinking cache): v2.1.85–101에 영향, 제 버전은 3월 26일~4월 10일 노출 (**15일**), v2.1.101에서 서버 측 수정
- 버그 #3 (≤25 words): v2.1.111+, 고정된 제 버전에는 해당 없음

세 가지 버그 중 하나가 저에게 영향을 미쳤습니다.
나머지 두 가지는 이미 그 아래 버전에 고정되어 있었기 때문에 해당하지 않았습니다.
이것은 버전 고정이 방어 전략으로서 유효함을 검증합니다 — 그리고 포스트모템 자체가 **Anthropic이 고지 없이 동작에 영향을 미치는 변경을 배포한다**는 것을 확인해 줍니다.
이것이 바로 버전 고정이 존재하는 이유입니다.

Opus 4.6은 최소 [2027년 2월 5일](https://platform.claude.com/docs/en/about-claude/model-deprecations)까지 활성 상태입니다.
10개월의 활주로가 있습니다.
포스트모템은 제 결정을 바꾸지 않았습니다 — 강화했습니다.

## 진짜 해결책: AI 개발 도구를 위한 공식 LTS

이런 일을 해야 할 이유가 없어야 합니다.
버전 고정, 프록시 모니터링, 수동 CHANGELOG 비교 — 이것들은 벤더가 구조적으로 해결해야 할 문제에 대한 사용자 측 방어입니다.

**AI 기반 CLI/IDE 도구를 배포하는 모든 회사에는 공식 LTS(Long-Term Support) 채널이 필요합니다.**
Anthropic만이 아닙니다 — Cursor, GitHub Copilot, Windsurf, Cline, 그리고 개발자의 워크플로우에 개입하는 모든 도구에 적용됩니다.

### 사용자 측 고정으로는 충분하지 않은 이유

제 셋업은 작동합니다: v2.1.109에 고정, `DISABLE_AUTOUPDATER=1` 설정, 프록시로 모니터링.
그러나 이 접근 방식은:

- **확장되지 않습니다.** 대부분의 사용자는 투명 프록시를 운영하거나 CHANGELOG를 읽지 않습니다. 그저 "나빠졌다"고 느끼면서 원인을 진단하지 못합니다.
- **적대적입니다.** 자신의 도구의 업데이트 메커니즘과 싸우고 있습니다. 도구는 능동적으로 최신 버전을 원합니다 — 자동 업데이터, 지원 중단 타이머, 현재 버전을 가정하는 기능 플래그.
- **만료 기한이 있습니다.** Opus 4.6은 2027년 2월에 은퇴합니다. 오래된 클라이언트 버전은 결국 API 비호환에 직면합니다. 고정된 버전이 계속 작동한다는 보장이 없습니다.
- **책임을 사용자에게 전가합니다.** 벤더가 브레이킹 체인지를 배포하면, 고정하지 않은 사용자가 고정하지 않은 것에 대해 비난받습니다. 이것은 본말전도입니다.

### 공식 LTS의 모습

이 모델은 모든 성숙한 소프트웨어 생태계에 이미 존재합니다 — Node.js, Ubuntu, PostgreSQL, Java.
AI 개발 도구의 경우:

| 특성 | 현재 현실 | LTS 채널 |
|---------|----------------|-------------|
| 업데이트 주기 | 지속적 (매일 릴리스) | 분기별 보안/안정성 패치만 |
| 모델 변경 | 언제든지, opt-in 없이 | LTS 기간 동안 모델 고정, 명시적 마이그레이션 경로 |
| 시스템 프롬프트 변경 | 보이지 않고, 기록되지 않음 | 문서화되고, LTS 사용자에게 opt-in |
| 브레이킹 동작 변경 | 고지 없이 배포 | 명시적 마이그레이션 요구 (deprecation → removal 사이클) |
| 최소 지원 기간 | "보다 빠르지 않게" 지원 중단 날짜 | LTS 릴리스로부터 **최소 12개월 보장** |
| 롤백 | 사용자 책임 (고정 + 프록시) | 벤더 제공: `claude --channel lts` 또는 동등한 방법 |

### 비즈니스 케이스

이것은 이타심이 아닙니다 — 리텐션입니다:

- **엔터프라이즈 신뢰.** 동작이 조용히 변경되는 도구에 의존할 엔터프라이즈는 없습니다. 공식 LTS = 엔터프라이즈 준비 완료. 이것 없이는 모든 조달팀이 "벤더 불안정성"을 리스크 매트릭스에 추가합니다.
- **지원 부하 감소.** Anthropic의 100건 이상의 Opus 4.7 이슈 중 절반은 사전 고지 없는 변경에 혼란스러운 사용자들입니다. 안정 채널은 이 카테고리를 완전히 제거합니다.
- **경쟁 차별화.** 진정한 LTS를 최초로 제공하는 AI 코딩 도구가 "프로덕션 인프라" 시장 세그먼트를 차지합니다. 현재 모든 도구는 데모 데이에 최적화되어 있지, 200일차에 최적화되어 있지 않습니다.

### 제가 요구하는 것

Anthropic에게, 그리고 AI 개발 도구를 배포하는 모든 회사에게:

1. **보장된 지원 기간(12개월 이상)의 공식 LTS 채널**
2. **LTS 채널에서 보이지 않는 동작 변경 금지** — 모든 변경 문서화, 은밀한 시스템 프롬프트 수정 금지
3. **모델 버전 보장** — LTS 사용자는 명시적으로 마이그레이션할 때까지 선택한 모델 유지
4. **롤백 메커니즘** — "버전 고정하고 기도하기"가 아닌, 벤더가 지원하는 채널 전환
5. **시스템 프롬프트 변경을 포함하는 투명한 CHANGELOG** — 클라이언트 코드 변경만이 아닌

현재 상황 — 사용자가 도구가 왜 저하되었는지 이해하기 위해 프록시 데이터를 역공학해야 하는 상황 —
은 벤더와 전문 개발자 사이의 지속 가능한 관계가 아닙니다.
